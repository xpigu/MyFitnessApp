# 健康页面拓展优化方案

## 1. 当前页面存在的问题

从软件设计和产品可用性的角度看，当前健康页面已经具备基础模块，但仍存在以下明显不足：

### 1.1 信息是分散的，缺少首页级决策视图

- 页面上有饮食、挑战、签到、训练等元素，但没有把这些内容整合成一个“今天健康状态如何”的统一结论。
- 用户打开页面后，需要自己理解多个卡片，缺少明确的优先任务和行动建议。

### 1.2 关键数据仍是静态占位，真实感不足

- 顶部“今日进度”中的步数、运动时长、卡路里存在静态展示问题。
- 饮食记录中三餐内容和热量明细大部分仍是示例数据，而不是完全来自当天真实记录。
- 页面会给人“像原型图而不是产品”的感觉。

### 1.3 缺少个性化表达

- 问候语、用户名、重点建议没有根据用户资料和当天状态动态调整。
- 页面没有形成“这是属于这个用户今天的健康首页”的感受。

### 1.4 缺少任务闭环

- 用户能看到补水、签到、训练这些能力，但页面没有告诉用户“下一步最值得做什么”。
- 没有形成“状态诊断 -> 推荐动作 -> 执行入口 -> 结果反馈”的闭环。

### 1.5 扩展性一般

- 当前布局里部分内容写死在 XML 中，后续如果要接入更多真实数据，会越来越难维护。
- 如果后面要增加健康评分、异常提醒、连续目标完成率等能力，现有结构不够灵活。

## 2. 优化目标

本轮优化不追求一次性把所有功能做满，而是先把健康页面升级成一个更像“个人健康仪表盘”的首页。

核心目标：

1. 让用户一眼看懂今天状态
2. 让页面展示尽量由真实数据驱动
3. 让页面给出清晰的下一步建议
4. 为后续继续接入健康评分、趋势分析、AI建议预留结构

## 3. 设计方案

### 3.1 新增“今日健康重点”卡片

设计目的：

- 在页面上方增加统一的健康摘要区域
- 汇总签到、补水、训练完成情况
- 根据当前状态自动生成一句话建议
- 提供最直接的下一步操作入口

建议展示内容：

- 当前状态标签：如“表现优秀”“继续推进”“今天还差一点”
- 一句话摘要：告诉用户今天最应该补什么
- 三项状态清单：签到、补水、训练
- 主按钮：根据当前缺口动态跳转到最需要执行的动作
- 次按钮：跳转统计页查看整体数据

### 3.2 顶部概览改为真实数据驱动

设计目的：

- 让“今日进度”从装饰性展示变成可信数据看板

拟接入的数据：

- 步数：优先使用跑步记录中的真实步数，没有则根据跑步距离做保守估算
- 卡路里：使用当天训练总消耗
- 锻炼：使用当天训练总时长

同时将圆环进度与用户目标关联：

- 步数对比 `targetDailySteps`
- 卡路里对比日常训练目标
- 运动时长按 60 分钟为默认日目标

### 3.3 饮食记录改为按当天真实数据渲染

设计目的：

- 去掉早餐、午餐、晚餐、加餐中的静态示例内容
- 每个时段展示当天真实记录
- 无记录时展示空状态

这样做的好处：

- 页面可信度更高
- 用户能立即看到自己今天到底记录了什么
- 后续可以继续扩展删除、编辑、推荐食物等能力

### 3.4 加强个性化表达

设计目的：

- 头部问候语根据当前时间变化
- 用户名从用户资料中读取
- 健康建议结合当天训练、饮水、签到和饮食状态生成

## 4. 实施步骤

### 第一步：梳理页面真实数据来源

- 用户资料：`UserProfileViewModel`
- 饮食记录：`DietRecordViewModel`
- 签到状态：`DailyCheckinViewModel`
- 训练汇总：`WorkoutRecordViewModel`

### 第二步：重构页面信息层级

- 在头部和今日进度之间新增“今日健康重点”卡片
- 让用户先看到摘要，再看明细模块

### 第三步：把顶部关键指标改成动态数据

- 替换静态数值
- 计算并展示对应进度

### 第四步：把饮食记录改成数据驱动渲染

- 按早餐、午餐、晚餐、加餐分组
- 渲染真实记录列表
- 无记录显示提示文案

### 第五步：补齐交互闭环

- 根据今日状态动态决定主操作按钮
- 可以直接去签到、去补水、去训练或查看统计

### 第六步：验证可维护性

- 检查页面是否仍可正常编译
- 检查新增逻辑是否引入明显诊断问题

## 5. 本轮执行内容

本次会优先完成以下可直接落地的优化：

1. 新增“今日健康重点”卡片
2. 顶部进度改为真实训练/饮食/用户目标驱动
3. 饮食记录改为当天数据渲染
4. 头部问候语与用户名动态化

## 6. 后续可继续扩展的方向

如果下一轮继续优化，建议按以下优先级推进：

1. 增加健康评分系统
2. 增加近 7 天趋势可视化
3. 加入异常提醒，如饮水不足、晚餐未记录、连续未训练
4. 将“活跃挑战”改造成真实挑战任务系统
5. 接入更完整的步数与传感器数据

## 7. 预期结果

优化完成后，健康页面应达到以下效果：

- 一眼可读：先给状态结论，再看详细数据
- 数据可信：核心指标尽量来自真实记录
- 可执行：用户知道此刻下一步该做什么
- 可扩展：后续继续加健康分析时不需要推翻重做
